# Claude Code 一人公司多角色协作基线

| 字段 | 值 |
|------|-----|
| 版本 | v1.0 |
| 状态 | 通用模板 |
| 适用范围 | 使用 Claude Code 的单人/小型项目 |

## 1. 目标

这套基线把 Claude Code 会话组织成一个轻量开发团队：同一个人可以在不同时间以 PM（产品经理）、UI（界面设计师）、Architect（架构师）、Developer（开发工程师）、Tester（测试工程师）、DevOps（运维/部署工程师）等角色启动 Agent，让项目拥有启动、需求、界面、设计、实现、测试、部署和复盘的持续记忆。

人类用户是项目 Owner（负责人）和实际项目经理，负责最终协调、优先级判断和流程取舍。工作流不设置常驻 Project Manager（项目经理）Agent，避免让一人公司产生没有实际价值的管理层。

它不是企业级流程管理系统。它的重点是：少量文件、明确职责、可追溯状态、可复用模板。

## 3. 角色名称

文档中首次出现角色时应优先使用“英文代号（中文名称）”格式，方便阅读，同时保留稳定角色 id。

| 英文代号 | 中文名称 |
|----------|----------|
| PM | 产品经理 |
| UI | 界面设计师 |
| Architect | 架构师 |
| Developer | 开发工程师 |
| Tester | 测试工程师 |
| DevOps | 运维/部署工程师 |
| Role Creator | 角色创建者 |

## 4. 语言规则

这套工作流默认且必须以中文进行对话和协作。

- Agent 与用户的沟通使用中文。
- PRD、设计文档、Review 结论、部署检查、角色日志、纠错记录和基线修正提案使用中文。
- 代码标识符、命令、错误信息、第三方 API 名称、英文引用和用户明确要求的外文内容可以保留原文。
- 如果用户要求输出英文或双语内容，以用户当次明确要求为准；否则回到中文。
- 不要因为角色名是 PM、Architect、Developer、DevOps，就把工作流对话切换成英文。

## 5. 工作模式

本工作流支持标准迭代和非迭代工作两类模式。标准迭代用于明确要落地的产品功能；Bugfix、线上故障、产品方案、UI 草案、技术预研、运维任务等不强制进入迭代。

具体模式选择、记录路径和升级规则见 `docs/baseline/work-modes.md`。

## 6. 团队知识库

团队知识库用于沉淀可复用知识、未落地机会、架构取舍、Bug 根因、部署经验和复盘结论。它不是流水账，不能替代角色日志。

知识库路径为 `docs/knowledge/`。写入规则、分类和读取规则见 `docs/baseline/knowledge-base.md`。

## 7. 上下文治理

随着迭代次数增加，角色日志和 Review 记录会快速膨胀。Agent 启动时只读索引、摘要和相关文件，不读完整历史。

日志摘要、归档触发条件和读取层级见 `docs/baseline/context-policy.md`。

## 8. 标准迭代流水线

Bootstrap 初始化不是标准迭代本身。它只安装团队工作台。只有用户明确选择标准迭代后，才进入下面的流水线。

```text
PRD 阶段 -> UI 方案阶段 -> 设计阶段 -> 实现阶段 -> 测试阶段 -> 部署就绪检查 -> 迭代关闭检查 -> 收尾归档
```

| 阶段 | 产出方 | Review 方 | 定稿条件 |
|------|--------|-----------|----------|
| PRD | PM（产品经理） | 产出方按影响领域动态指定 | 指定 Review 方全部通过 |
| UI 方案 | UI（界面设计师） | 产出方按影响领域动态指定 | 指定 Review 方全部通过，或明确本迭代无 UI 变更 |
| 设计 | Architect（架构师） | 产出方按影响领域动态指定 | 指定 Review 方全部通过 |
| 实现 | Developer（开发工程师） | 产出方按影响领域动态指定 | 指定 Review 方全部通过 |
| 测试 | Tester（测试工程师） | 产出方按影响领域动态指定 | 测试报告通过，阻塞缺陷关闭或明确延期 |
| 部署 | DevOps（运维/部署工程师） | 默认独立检查；高风险时动态指定确认方 | 部署检查通过，或明确记录阻塞/跳过原因 |
| 迭代关闭 | 当前 Agent 执行关闭检查 | 用户确认 | INDEX、迭代记录、角色日志、Change Note、知识沉淀状态一致 |

当前阶段未定稿前，不进入下一阶段。Spike 或纯技术预研可以跳过部分阶段，但必须在迭代记录中说明原因和实际 Review 方。

PM（产品经理）负责产品内容是否正确，但不代表人类做最终项目协调。阶段推进、优先级取舍、是否接受风险延期，由人类 Owner 最终确认。

Bootstrap 初始化、收尾归档、迭代关闭检查、流程审计不是角色，而是非角色机制。触发时机、执行者和结束条件见 `docs/baseline/mechanisms.md`。任何当前会话 Agent 都可以在用户要求或自动检测到触发条件后执行这些机制，但关键结果必须由用户确认。

## 9. 状态机

产出文档和阶段门禁使用同一套状态词：

| 状态 | 含义 |
|------|------|
| 待Review | 初版刚提交，尚无人 Review |
| Review中 | Review 方正在反馈，或等待部分 Review 方反馈 |
| 修改中 | 产出方正在处理 Review 问题 |
| 已定稿 | 当前轮次指定 Review 方均通过 |
| 阻塞 | 需要人类或外部条件解除 |
| 已跳过 | 该阶段不适用于本迭代，必须写原因 |

状态真源规则：

- `docs/progress/iterations/vX.Y.md` 是迭代阶段门禁和 Review 轮次的真源。
- `docs/progress/INDEX.md` 是项目级当前状态和下一步入口的真源。
- `docs/baseline/project-context.md` 只写项目事实，不写当前阶段或 Review 状态。
- PRD、设计文档、测试报告头部可以保留状态摘要，但定稿前必须检查它们和迭代记录一致。
- 任何角色改变阶段状态时，必须先更新迭代记录，再同步 `INDEX.md`；不要只改自己的产出物。

## 10. 动态 Review 计划

标准迭代不采用全员固定 Review。每个产出方在提交产出物时，必须根据本次变更影响领域动态指定 Review 方，并写明指定理由和未指定理由。

### 10.1 选择维度

| 影响领域 | 应考虑的 Review 方 |
|----------|--------------------|
| 产品范围、用户故事、验收标准 | PM（产品经理） |
| 用户流程、页面、交互、视觉状态 | UI（界面设计师） |
| 架构、数据流、接口、技术选型 | Architect（架构师） |
| 实现复杂度、工程成本、兼容性 | Developer（开发工程师） |
| 验收标准、边界条件、回归风险 | Tester（测试工程师） |
| 部署方式、环境变量、云服务、发布风险 | DevOps（运维/部署工程师） |

例如：全新项目通常应指定 Architect（架构师）；全新 UI 应指定 UI（界面设计师）；部署方式从本地改为云端，应指定 DevOps（运维/部署工程师）。

### 10.2 最低质量底线

- 标准迭代的核心产出默认至少指定 2 个 Review 方。
- 如果少于 2 个 Review 方，产出方必须写明原因，并由用户确认。
- Review 方不是越多越好；只指定真正受影响或能提供关键判断的角色。
- 定稿条件是：本轮 Review 计划中指定的 Review 方全部完成并通过，且无未处理的高严重度阻塞问题。

### 10.3 Review 计划模板

```markdown
## Review 计划
- Review 模式：动态 Review
- 影响领域：
- 本轮指定 Review 方：
  - 角色：原因
- 未指定角色及原因：
  - 角色：原因
- 少于 2 个 Review 方时的用户确认：（如不适用写“无”）
- 定稿条件：本轮指定 Review 方全部通过，且无未处理高严重度阻塞问题
```

## 11. Review 轮次

每个阶段按 `R1`、`R2`、`R3` 递增。

- `R1`: 初版提交。
- 有任何 `需修改`：产出方进入 `修改中`，修完后提交 `R2`。
- 本轮指定 Review 方全部 `通过`：产出方将阶段改为 `已定稿`。

Review 记录必须包含：轮次、结论、问题、严重程度、建议。不要只写“感觉不行”。

## 12. 定稿后问题处理

文档定稿后，未参与 Review 的角色或后续执行角色可能发现问题。原则上不直接修改已定稿正文，而按影响程度分三档处理。

| 档位 | 适用情况 | 处理方式 |
|------|----------|----------|
| 实现取舍 | 方案可实现，但不是最优；不改变产品范围、接口契约、UI 验收或部署方式 | 不改定稿文档，在实现记录或角色日志中写明取舍，必要时沉淀到知识库 |
| 轻量变更 | 需要补字段、补边界说明、调整小范围验收或接口细节，但不改变阶段大方向 | 创建 Change Note，由受影响角色确认，不重启完整 Review |
| 重大变更 | 产品范围、主流程、架构方案、部署方式、关键验收发生变化 | 创建变更提案，回到对应阶段重新 Review |

### 12.1 Change Note 流程

Change Note 不是完整 Review 的重启，也不是绕过 Review 的后门。它只处理已定稿内容中的轻量补充、实现中发现的小范围事实修正、以及受影响角色能够快速判断的变更。

状态流转：

```text
待确认 -> 已确认 -> 已执行 -> 已归档
```

特殊状态：

```text
已废弃 / 升级为重大变更
```

执行规则：

1. 提出角色创建 Change Note，说明问题、影响范围、建议处理方式和需要确认的角色。
2. 需要确认的角色由提出角色按影响领域动态判断，不照固定表机械执行。
3. 被点名的受影响角色可以补充“还需要谁确认”或“我不受影响”的理由。
4. 受影响角色全部确认后，Change Note 进入 `已确认`，可以直接修改文档、代码或开发方案。
5. 如果执行中发现影响范围扩大，立即停止轻量变更，改为 `升级为重大变更`。
6. 执行完成后，提出角色或执行角色补充执行记录、验证证据、关联提交，并改为 `已执行`。
7. 收尾归档时，将 Change Note 登记到迭代记录或 ad-hoc 记录；如修改已定稿文档，文档中必须保留 Change Note 引用。
8. 完成索引、日志和知识沉淀后，状态改为 `已归档`。

Change Note 完成条件是：本次指定的受影响角色已确认，执行记录和验证证据已补齐，相关索引已更新。未指定的角色后续再提出问题时，按本节三档规则重新判断，不回头假装原文档没有定稿过。

### 12.2 定稿后问题的处理底线

- 不能静默修改已定稿正文。
- 不能因为“只是 Agent 自己实现起来麻烦”就重开完整 Review。
- 不能把重大变更伪装成轻量变更。
- 允许实现角色在不改变契约的前提下做实现取舍，但必须写入实现记录或角色日志。
- 允许轻量变更在受影响角色确认后直接执行，但必须可追溯、可回看、可归档。

## 13. 写权限

| 文件 | 谁能改正文 | 谁能追加 |
|------|------------|----------|
| PRD | PM（产品经理） | Review 计划中指定的角色追加 Review |
| 设计文档 | Architect（架构师） | Review 计划中指定的角色追加 Review |
| 代码 | Developer（开发工程师） | Review 方不直接改，除非用户明确要求 |
| UI 方案 | UI（界面设计师） | Review 计划中指定的角色追加 Review |
| 测试报告 | Tester（测试工程师） | Review 计划中指定的角色追加 Review |
| Change Note | 提出变更的角色或被用户指定的执行角色 | 受影响角色确认、执行角色补充证据 |
| 非迭代工作记录 | 执行角色 | 其他角色只在被邀请时追加意见 |
| 团队知识库 | 各角色按领域维护 | 其他角色可补充来源和交叉引用 |
| 迭代记录 | 各阶段相关角色按职责更新 | Review 方追加自己的 Review 记录 |
| 角色日志 | 对应角色 | 无 |
| baseline/ | 经过人类确认后指定角色修改 | 任何角色可提出提案 |

## 14. Git 安全规则

每次开始：

```bash
git rev-parse --is-inside-work-tree
```

如果是 Git 仓库：

```bash
git status --short
git remote get-url origin
git log --oneline -10
```

如果配置了远端，再执行：

```bash
git pull --rebase
```

如果不是 Git 仓库，不执行 `git status`、`git pull` 或 `git log`。Bootstrap 阶段询问用户是否初始化 Git。

如果发现未归属修改，先判断是否属于本次任务。不能覆盖、重置或顺手清理他人修改。

每次结束：

如果是 Git 仓库：

```bash
git status --short
git add <本次文件>
git commit -m "[Role] vX.Y 动作摘要"
```

如果配置了远端，再执行：

```bash
git pull --rebase
git push
```

如果不是 Git 仓库，记录“未提交：当前目录不是 Git 仓库”，不要运行 Git 提交命令。

实现阶段 Review 必须记录 diff 范围：`base_commit` 到 `head_commit`。

## 15. 角色日志和纠错

每个角色维护：

```text
docs/progress/roles/{role}.md
docs/progress/roles/{role}-corrections.md
```

角色日志记录本次做了什么。纠错记录记录本角色犯过什么流程错误、如何避免复发。纠错记录保持 30 条以内。

角色日志超过上下文阈值时，必须按 `docs/baseline/context-policy.md` 做摘要归档。长期有价值的信息应提炼到 `docs/knowledge/`，不要只埋在日志里。

## 16. 基线修正

任何角色发现流程规则有问题，不能直接改 `baseline/`，先在角色日志中写：

```text
[基线修正提案] 问题：...；建议：...
```

Architect（架构师）或 Role Creator（角色创建者）可汇总提案，但修改基线必须经过人类确认。

## 17. 角色创建机制

角色创建是可行的，但必须受控。坏的角色会增加流程噪音，让一人公司变成“一个人在管理一堆虚假岗位”。

新增角色必须满足至少一条：

- 有独立产出物。
- 有独立 Review 视角。
- 有明确不可替代的专业边界。
- 能减少现有角色的认知负担。

不应该新增角色的情况：

- 只是把一个任务换个名字。
- 职责和现有角色高度重叠。
- 没有产出路径、状态入口或退出条件。
- 增加的协调成本大于收益。

新增角色必须通过 `role-creator.md` 的流程，并更新角色矩阵、迭代模板、日志和纠错文件。
